title: git协作小记
date: 2017-05-06 12:07:50

tags: git

一直以来，我在前几家公司的git协作方式都是使用git分支来协作的：以master分支作为线上部署代码来源，通过master上打tag，以打的tag的版本号来作为发布版本。
开发的都是以develop作为开发分支，类似gitflow的协作模型，各个开发从develop分支checkout出新分支，比如 feature/someFeature，或者bugfix/bugMum等，当然不一定完全遵守gitflow模型，总体思路是一致的。合并的时候一般对分支作权限，比如一般开发没有往develop的push或者merge权限，需要在本地自己的开发分支先merge develop分支代码，然后发起一个pull request（gitlab为merge request）这样项目负责人accept pr的时候就不会发生冲突。

最近在新公司采取的是另外一种git协作模式，采用的fork协作。就是对源代码fork一份代码出来，然后本地自己开发，同时需要sync源库的代码，需要

两种协作模式都有很多人用，我个人比较喜欢前一种方式，应为我觉得git的精髓就是它的分支特性，git的分支协作可以生成一个结构完整的树状git commit的log记录。fork更适合开源项目的协作方式。（我还不知这种git协作方式的git log树会是怎么样的，是否与使用分支模式一样？）
